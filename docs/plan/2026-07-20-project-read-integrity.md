# CoreStudio 项目读取完整性修复计划

**日期：** 2026-07-20  
**状态：** 执行中

## 目标

在不删除图片资产、不改变项目格式版本的前提下，补齐 CoreStudio 的项目读取边界，让损坏、旧版或局部异常的数据不能再把整个项目打开流程拖垮；同时修复 Excalidraw 基线文档与机器记录不一致的问题，防止后续升级从错误基线开始。

本轮继续遵守四条原则：

1. 旧数据兼容读取，新写入数据严格校验；
2. 单条坏数据隔离，不能扩大成整个项目不可用；
3. 自动修复只处理确定性问题，原始数据和图片资产不删除；
4. 所有容错路径都要有可定位的诊断信息，不能静默吞错。

## PR 1：基线事实来源收口

### 范围

- 把 `docs/doc/excalidraw-fork-maintenance.md` 的“当前源码基线”更新为 `excalidraw/upstream-baseline.json` 中的 `currentSha`；
- 将初始导入基线与当前升级基线明确分开，避免以后继续从旧提交计算差异；
- 以 `upstream-baseline.json` 的补丁分组作为当前补丁清单事实来源；
- 新增合同测试，确保维护文档记录的当前 SHA 始终与 JSON 一致。

### 完成标准

- 合同测试能在文档 SHA 漂移时失败；
- 当前 SHA、初始导入 SHA 和后续升级入口表述清楚；
- 桌面测试和类型检查通过。

## PR 2：项目读取边界

### 范围

#### `project.json`

- 用显式解析器代替直接类型断言；
- 拒绝非对象和不受支持的未来格式；
- 对旧版缺失字段做确定性规范化，固定项目内部文件名，限制路径字段不能越出项目目录；
- 读取时返回诊断，写入前只接受规范化后的合法 manifest。

#### `recent-projects.json`

- 校验数组中的每一条记录；
- 跳过坏记录，保留其他可用记录，并继续执行去重和不存在路径清理；
- 整个文件无法解析时不阻断应用启动，也不覆盖原始损坏内容。

#### `scene.excalidraw`

- 在进入 renderer 前完成 JSON 和顶层结构预检；
- 将语法错误、结构错误和不支持格式区分成可操作错误；
- 安全模式不伪造空画布，不允许一次自动保存覆盖仍可人工恢复的原始场景。

#### 图片写回 journal

- 用显式解析器校验 journal；
- 单个损坏 journal 隔离处理，不能阻断其他事务恢复和项目打开；
- 不把解析失败等同于事务已完成，不删除对应原始文件或图片资产。

### 完成标准

- 先增加失败测试，再完成最小实现；
- 覆盖非对象、缺字段、错误字段、混合好坏记录、损坏 JSON 和损坏 journal；
- 已有旧项目读取、图片记录修复、事务恢复测试继续通过；
- 完整桌面测试、类型检查、secret scan 和 production build 通过。

## 本轮不做

- 不调整图片读取并发；该项需要先用大项目测量峰值内存和打开耗时，再确定并发上限；
- 不重写发布流程；发布命令收口作为独立工程化任务处理；
- 不改 ACP、`activeTool` 或内置生成产品逻辑；
- 不升级项目格式版本，不迁移或清理图片资产。

## 执行顺序

1. PR 1 修复基线文档并建立自动防漂移合同；
2. PR 2 依次补齐 manifest、最近项目、scene、journal 的失败测试和实现；
3. 执行完整验证，通过受保护 PR 合入；
4. 另开任务对图片并发和发布命令做测量与方案设计。
