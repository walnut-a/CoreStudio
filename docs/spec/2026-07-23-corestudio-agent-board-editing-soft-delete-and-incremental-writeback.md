# CoreStudio Agent Board 画布编辑、软删除与增量写回需求整理

> 所属项目：CoreStudio
> 需求状态：需求已确认，待技术方案与开发
> 首版范围：V1
> 日期：2026-07-23

## 需求背景

CoreStudio 本体鼓励用户在画布中使用线条、箭头、方块、文字和自由笔整理思路。这些内容既是设计过程的一部分，也是模型理解图片关系、设计意图和修改方向的重要上下文。

Agent Board 已经可以在 Codex 内置浏览器中打开项目画布、选择元素、复制固定引用，并把当前运行态画布暴露给 Codex。但当前 Agent Board 的画布修改采用 `runtime-only` 策略：

- 用户可以在浏览器里新增、移动、修改或删除元素；
- 修改会立即反映在当前浏览器画面和运行态上下文中；
- 修改不会保存回 CoreStudio 项目；
- 页面刷新、切换项目或桌面端产生新版本后，浏览器中的修改可能消失。

这会产生明显的认知冲突：界面看起来是一张可编辑画布，但实际只保存选择和运行态上下文。尤其是删除图片时，用户会看到图片消失，却无法确认它是否已经同步到 CoreStudio 本体。

本需求要让 Agent Board 保留完整的画布思考能力，同时避免浏览器使用旧场景覆盖桌面端或 Codex 的新写入。

## 当前理解

- 目标用户：在 Codex 中打开 CoreStudio 内置画布，结合图片、图形和文字整理设计思路的用户。
- 使用场景：用户查看项目图片、补充标注、调整元素、删除不需要的画布内容，并继续让 Codex 分析、生成或写回结果。
- 需求目标：让 Agent Board 中合理的画布编辑成为真实项目修改，同时确保 CoreStudio 仍是项目数据所有者，避免整张旧画布覆盖最新项目。

## 产品目标

1. 保留 Agent Board 中画线、画箭头、画图形、写文字和自由绘制等思考能力。
2. 允许用户正常新增、移动、修改和删除项目元素，不把 Agent Board 降级成只读预览。
3. 让 Agent Board 的有效编辑同步到 CoreStudio 本体，刷新或重新打开后仍然存在。
4. 通过元素级增量写回控制与桌面端、Codex 同时编辑时的冲突范围。
5. 画布删除始终采用软删除，不因删除画布元素而删除项目中的图片资产。
6. 修正项目健康检查和修复逻辑，避免把用户主动删除的图片重新补回画布。

## 不在本需求内

- 物理删除项目中的原始图片资产。
- 自动清理未被画布引用的图片资产。
- 多人实时光标、在线协作房间或 CRDT 级实时合并。
- 让 Agent Board 直接调用 CoreStudio 内置生图模型。
- Agent Board 图片导入链路的重做。
- Codex 任务状态、运行进度或 thread 信息在画布中的展示。
- 用 Agent Board 的整张场景快照覆盖项目文件。

## 名词定义

| 名称         | 定义                                                                     |
| ------------ | ------------------------------------------------------------------------ |
| 项目场景     | CoreStudio 项目中持久化的 Excalidraw 元素集合及必要场景状态。            |
| 运行态画布   | Agent Board 当前浏览器内存中的画布状态，可包含尚未完成写回的本地变化。   |
| 元素级增量   | 针对明确元素 ID 的新增、更新、软删除或恢复操作，不包含整张场景覆盖。     |
| 软删除       | 将元素标记为已删除并保留元素 ID、版本、图片记录和原始资产。              |
| 图片资产     | 项目文件夹中由图片记录指向的原图文件，以及可重新生成的缩略图和预览缓存。 |
| 已提交资产   | 已完成图片写回事务并正式纳入项目图片记录的资产。                         |
| 写回事务回滚 | 图片写入尚未提交成功时，为恢复事务前状态而撤销本次新增记录和临时资产。   |
| 项目版本     | 用于判断浏览器编辑基于哪一份项目场景的版本或内容哈希。                   |

## 产品原则

1. **CoreStudio 仍是项目数据所有者。** Agent Board 不直接修改项目文件，只能通过 Local Bridge 请求 CoreStudio 应用修改和保存。
2. **允许正常编辑，但禁止整场景覆盖。** 用户可以修改项目元素；浏览器只能提交元素级增量，不能提交一份旧场景替换最新项目。
3. **删除元素不等于删除资产。** 已提交图片资产不随画布元素删除而删除。
4. **用户意图优先于自动修复。** 有明确软删除记录的图片不得被健康检查当成异常丢失，也不得被项目修复自动补回。
5. **不同元素尽量自动合并，同一元素不静默覆盖。** 并发编辑发生在不同元素时正常合并；发生在同一元素时必须识别冲突。
6. **选区和视口仍是临时上下文。** 选中元素、画布视口、状态条开关和 Codex 任务状态不写入项目场景。
7. **事务回滚不是用户删除。** 尚未提交成功的新图片写入可以在事务回滚时清理；这不改变已提交资产长期保留的规则。

## V1 功能范围

### 1. Agent Board 保留的画布能力

Agent Board 应继续开放以下能力：

- 选择一个或多个元素；
- 平移、缩放和浏览画布；
- 新增矩形、菱形、圆形等基础图形；
- 新增直线和箭头；
- 使用自由笔绘制；
- 新增和编辑文字；
- 移动、缩放、旋转已有元素；
- 修改元素颜色、线型、透明度等常规属性；
- 删除和恢复已有元素；
- 使用撤销和重做处理上述操作。

这些操作不只用于视觉整理，也应成为 Codex 可以读取的项目上下文。

#### 1.1 元素编辑侧栏

##### 当前问题

CoreStudio 当前通过 Excalidraw 的 `renderSelectedShapeActions` 把原生元素操作内容放进右侧“详情”栏。原生操作内容会根据画布尺寸在完整、紧凑和移动布局之间切换。

当 Agent Board 的可视区域被识别为平板或紧凑模式时，右侧栏会收到一组原本为浮动工具条设计的无文字图标，并按纵向排列。用户看到的是间距较大的图标列，难以判断当前属性、按钮含义和可执行操作。这不是元素数据异常，而是原生紧凑工具条与固定详情侧栏的布局模型不匹配。

##### 目标形态

- 右侧“元素编辑”使用固定的侧栏型布局，不再跟随 Excalidraw 的完整、紧凑或移动工具条模式切换。
- 同一窗口宽度下，打开或关闭右侧栏不得导致元素编辑控件突然切换成另一种产品形态。
- 属性使用“名称 + 当前值/控件”表达；仅依赖图标的操作必须提供清晰文字、标题或稳定的悬浮说明。
- 相关属性按分组排列，优先保证可理解性，不把浮动工具条的所有按钮原样堆进侧栏。
- 侧栏可以继续复用 Excalidraw 的动作、可用性判断和状态源，但需要单独的 CoreStudio 侧栏渲染层，不能直接复用 `CompactShapeActions` 的布局。
- 创建元素仍由顶部工具栏负责；右侧栏只编辑当前选中元素，不重复放置创建工具。

##### 不同元素的编辑内容

| 选中状态     | 元素编辑侧栏                                                       |
| ------------ | ------------------------------------------------------------------ |
| 无选区       | 显示“选择画布元素后查看可编辑属性”，不保留空按钮占位。             |
| 单张图片     | 显示图片相关属性和操作，例如透明度、图层、裁剪、复制、软删除。     |
| 单个图形     | 显示填充、描边、线宽、线型、圆角、透明度、图层、复制和软删除。     |
| 单条线或箭头 | 显示描边、线宽、线型、箭头端点、透明度、图层、复制和软删除。       |
| 单段文字     | 显示字体、字号、对齐、颜色、透明度、图层、复制和软删除。           |
| 多选         | 只显示可以安全批量应用的公共属性、对齐、分布、分组、复制和软删除。 |
| 混合选区     | 不显示只适用于单一元素类型的控件；公共属性的混合值需要明确表达。   |

实际可见项继续以 Excalidraw 对当前选区的能力判断为准。表中列举的是产品分组，不要求为不支持的元素强行补齐属性。

##### 操作与保存

- 侧栏修改必须使用与画布直接操作相同的元素级增量写回链路。
- 用户点击复制、软删除、图层调整或修改属性后，不能只改变浏览器运行态。
- 保存中、保存失败和编辑冲突需要使用统一的画布保存反馈，不在每个属性控件旁重复展示状态。
- 在元素级增量写回尚未具备时，不得让侧栏表现成“已经永久修改项目”；开发阶段可以隐藏未闭环操作，不能静默保留为仅运行态操作。
- CoreStudio 桌面模式和 Agent Board 应共享属性含义和动作结果；可以采用不同布局，但不能形成两套编辑规则。

### 2. Agent Board 隐藏的菜单能力

Agent Board 中隐藏以下当前不可用或容易造成误解的主菜单项：

- 导出图片；
- 在画布上查找；
- 帮助；
- 重置画布。

CoreStudio 桌面模式继续保留这些菜单项。Agent Board 当前仍保留项目切换和主题切换。

“重置画布”属于项目级破坏操作，不应通过普通元素增量链路开放。是否需要新的项目级清空能力，后续单独讨论。

### 3. 元素级增量写回

#### 3.1 支持的操作

Agent Board 至少需要支持以下持久化操作：

| 操作       | 说明                                               |
| ---------- | -------------------------------------------------- |
| 新增元素   | 新增图形、线条、箭头、自由笔、文字等元素。         |
| 更新元素   | 修改位置、尺寸、旋转、样式、文字内容、绑定关系等。 |
| 软删除元素 | 将指定元素标记为已删除，不删除图片资产。           |
| 恢复元素   | 撤销软删除，恢复原元素 ID 并更新元素版本。         |

#### 3.2 提交原则

- 浏览器只提交发生变化的元素和必要的关联元素。
- 每个提交必须绑定项目 ID、项目版本和唯一操作 ID。
- 更新、删除和恢复必须携带提交方看到的元素版本。
- 箭头绑定、容器文字、分组等关联元素应作为同一操作批次原子提交，避免只保存一半关系。
- 连续拖动、缩放、自由绘制过程中不逐帧写回；在一次用户操作结束后提交稳定结果。
- 同一操作 ID 重试时不得重复创建元素或重复执行删除。
- 写回成功后，CoreStudio 返回新的项目版本和最终元素状态。

具体接口名、字段名和批处理窗口由技术方案确定，本文不提前固定。

### 4. 软删除与图片资产规则

#### 4.1 元素软删除

- 用户删除元素时，保留元素 ID，并将元素状态更新为软删除。
- 软删除需要更新元素版本和更新时间。
- 软删除元素默认不显示在画布上，也不进入 Codex 的当前可见画布和实时选区。
- 撤销删除时恢复同一个元素，不创建一个没有关系的新元素。
- 删除非图片元素时同样使用 Excalidraw 的软删除语义。

#### 4.2 图片资产保留

删除图片元素后，必须保留：

- 项目 `assets/` 中的原始图片；
- 图片记录；
- 生成来源、模型、提示词和时间等元数据；
- 父子图片关系和提示词引用关系；
- 软删除的画布元素记录。

普通项目保存、打开、版本同步、健康检查和项目修复都不得因为图片当前不在画布上而删除原图。

缩略图和预览图属于可重建缓存，可以继续由独立缓存清理逻辑处理；缓存清理不得延伸为原图清理。

#### 4.3 事务回滚例外

如果图片写回事务尚未成功提交，本次事务新创建的图片记录和资产可以在回滚时清理，以恢复事务开始前状态。

产品规则以“是否已经成功纳入项目”为边界：

- 已提交资产：画布删除后继续保留；
- 未提交事务资产：事务失败或回滚时允许清理。

### 5. 项目图片状态与本体规则调整

CoreStudio 本体当前的软删除和场景保存方式可以继续沿用，但项目健康检查、图片状态解释和项目修复需要认识软删除。

图片在画布中的状态至少区分为：

| 状态                    | 判断                                                 | 产品含义                           |
| ----------------------- | ---------------------------------------------------- | ---------------------------------- |
| `on-board`              | 存在未删除的图片元素                                 | 图片当前显示在画布上。             |
| `removed-from-board`    | 不存在未删除元素，但存在对应软删除元素               | 用户主动从画布移除，不是数据异常。 |
| `referenced-by-result`  | 图片本身不显示，但仍被其他有效结果引用               | 可以通过关联结果理解和定位。       |
| `missing-board-element` | 既没有有效元素，也没有软删除痕迹，且无法通过结果定位 | 可能是异常丢失，可以进入修复判断。 |

本体需要同步调整：

- `removed-from-board` 不计入孤立图片警告；
- 项目修复不得为 `removed-from-board` 创建新的图片元素；
- 生成记录和图片详情保留该图片，并显示“已从画布移除”一类明确状态；
- 从生成记录定位软删除图片时，不把它解释为项目损坏；
- 只有 `missing-board-element` 且资产、记录有效时，才允许自动补回画布；
- 同一个 `fileId` 仍有其他未删除图片元素时，图片状态继续视为 `on-board`。

### 6. 并发与冲突

#### 6.1 不同元素

桌面端、Codex 和 Agent Board 修改不同元素时，CoreStudio 应基于最新项目合并这些元素级操作，不要求用户处理整场景冲突。

示例：

- 桌面端移动图片 A；
- Agent Board 新增箭头 B；
- Codex 写回图片 C；
- 三项修改可以依次合并进入同一项目。

#### 6.2 同一元素

两个入口同时修改同一元素时，不使用静默的最后写入覆盖：

- 元素版本与请求预期一致：正常应用；
- 元素已经被其他入口更新：拒绝旧操作并返回冲突；
- 一个入口更新、另一个入口删除同一元素：按同元素冲突处理；
- 浏览器收到冲突后重新读取该元素和最新项目版本，并明确提示用户。

V1 不要求自动合并同一元素的不同字段。

#### 6.3 浏览器刷新

项目产生新版本后，Agent Board 应更新项目主体，但不能无条件整页重建并清除用户正在进行的操作。

- 已成功写回的 Agent Board 编辑必须保留；
- 未提交操作需要明确处于保存中或保存失败状态；
- 当前选区和视口应尽量保留；
- 如果关联元素因外部更新发生冲突，以服务端最新项目为准并提示用户。

### 7. 撤销与重做

- 撤销和重做不能只改变浏览器内存，必须产生对应的元素级反向操作并写回项目。
- 撤销删除时恢复原元素和原图片资产引用。
- 撤销新增时对新元素执行软删除，不删除图片资产。
- 撤销更新时需要基于最新元素版本提交；同一元素已被其他入口修改时按冲突处理。

### 8. Codex 上下文

- Codex 继续优先读取用户当前选区和固定引用快照。
- 软删除元素默认不进入当前选区、可见画布摘要和图片路径解析结果。
- Agent Board 新增的图形、箭头和文字在写回后，与桌面端创建的普通元素使用同一读取规则。
- 选区、视口和固定引用仍是运行态或任务级上下文，不因画布编辑持久化而改成项目任务数据。
- 画布只需要展示自身的保存中、已保存或保存失败状态，不需要反向获取 Codex 的任务运行状态。
- 如果 Codex 读取到了尚未写回的运行态元素，返回结果必须能区分项目已提交状态与浏览器待保存状态，不能把未保存元素静默当成稳定项目事实。

## 关键路径

### 新增标注

1. 用户在 Agent Board 中画箭头、图形或文字。
2. 浏览器完成一次操作后形成元素级增量。
3. Local Bridge 将增量交给 CoreStudio。
4. CoreStudio 基于最新项目版本应用增量并原子保存。
5. Agent Board 收到新项目版本并显示已保存。
6. CoreStudio 本体和 Codex 后续读取都能看到该元素。

### 修改已有元素

1. 用户移动、缩放或修改已有元素。
2. 浏览器提交元素 ID、预期版本和修改后的元素状态。
3. 元素未被其他入口更新时，CoreStudio 保存修改。
4. 元素已经变化时，CoreStudio 返回冲突，浏览器加载最新元素并提示用户。

### 删除图片

1. 用户在 Agent Board 中删除图片元素。
2. 浏览器提交软删除操作。
3. CoreStudio 更新元素的删除状态和版本。
4. 图片从 Agent Board 和 CoreStudio 本体画布消失。
5. 原图、图片记录和生成关系继续保留在项目中。
6. 项目健康检查不把该图片列为需要补回的异常。

## 状态与反馈

Agent Board 至少需要区分：

| 状态        | 行为                                                                           |
| ----------- | ------------------------------------------------------------------------------ |
| 已保存      | 当前已完成的画布编辑已写回项目。                                               |
| 保存中      | 正在提交一次稳定的元素操作，允许继续浏览；是否允许继续编辑由技术方案保证顺序。 |
| 保存失败    | 保留浏览器当前变化并明确提示，不得假装已经写回项目。                           |
| 编辑冲突    | 指定元素已经被其他入口更新，加载最新状态并提示用户重新操作。                   |
| Bridge 断开 | 不能把新编辑标记为已保存；断开期间是否允许继续临时编辑列入待确认。             |

这些状态只表达画布数据保存，不表达 Codex 的思考、生成或任务进度。

## 兼容与迁移

- 现有项目中的软删除元素继续有效，不需要升级原始图片资产。
- 现有没有软删除痕迹的孤立图片记录，仍按当前异常修复规则处理。
- 健康检查升级后，可能减少“孤立图片记录”警告数量，这是符合预期的规则修正。
- 不改变现有图片记录和原图路径格式。
- 元素级写回能力需要版本化的 Local Bridge 契约；旧版桌面端不支持时，Agent Board 不得退回整场景覆盖。

## 验收标准

### 正常编辑

- [ ] 在 Agent Board 新增图形、箭头、线条、自由笔和文字后，CoreStudio 本体可以看到相同元素。
- [ ] 移动、缩放、旋转和修改元素样式后，刷新 Agent Board 仍保留结果。
- [ ] Agent Board 不通过整张场景覆盖保存项目。
- [ ] 选择和视口变化不触发项目保存，也不更新项目版本。

### 软删除

- [ ] 在 Agent Board 删除图片后，CoreStudio 本体同步不再显示该元素。
- [ ] 被删除元素保留原元素 ID，并记录软删除状态和新版本。
- [ ] 图片原图仍存在于项目文件夹。
- [ ] 图片记录、生成元数据和关系数据继续存在。
- [ ] 撤销删除可以恢复原元素，不需要重新导入图片。
- [ ] 项目健康检查不把软删除图片列为需要补回的异常。
- [ ] 执行项目修复不会把软删除图片重新放回画布。

### 并发

- [ ] 桌面端和 Agent Board 修改不同元素时可以正常合并。
- [ ] Codex 写回新图片时不会覆盖 Agent Board 已保存的图形或文字。
- [ ] 两个入口修改同一元素时不会静默覆盖。
- [ ] 删除与更新发生在同一元素时能够识别冲突。
- [ ] 重复提交同一个操作 ID 不会重复创建或重复删除元素。

### 资产

- [ ] 普通删除和项目保存不会删除已提交原图。
- [ ] 缓存清理只处理可重建缓存，不处理原始图片资产。
- [ ] 未提交图片写回事务回滚时仍可清理本事务创建的临时资产。

### 界面边界

- [ ] Agent Board 保留常用绘图和元素编辑工具。
- [ ] 右侧元素编辑使用稳定的侧栏型布局，不随平板或紧凑断点变成无文字纵向图标列。
- [ ] 图片、图形、线条、箭头、文字、单选、多选和混合选区只显示适用的属性与操作。
- [ ] 无选区时显示明确空状态，不显示没有含义的按钮占位。
- [ ] 元素属性和操作具有名称、当前值或清晰说明，不能只依赖难以识别的图标。
- [ ] 通过右侧栏产生的修改使用同一元素级增量写回链路，刷新后仍然存在。
- [ ] Agent Board 隐藏导出图片、在画布上查找、帮助和重置画布。
- [ ] CoreStudio 桌面模式不受上述菜单隐藏影响。
- [ ] 画布保存状态不混入 Codex 任务状态。

## 待技术方案确认

1. 元素增量接口的最终命名、字段结构和 Bridge 版本号。
2. 一次用户操作结束的识别方式，以及自由绘制、文字编辑和连续拖动的提交时机。
3. 关联元素批次的原子边界和冲突返回结构。
4. Bridge 断开期间是禁止新增编辑，还是保留本地待提交队列。
5. Agent Board 读取到待保存运行态时，Codex 上下文如何携带明确的持久化状态。
6. 保存状态提示放在选区状态条附近还是独立的轻量位置。
7. 如何复用 Excalidraw 的动作和选区能力判断，同时避免继续继承 `CompactShapeActions` 的工具条布局。

以上问题属于技术设计或交互细化，不改变已经确认的产品原则：正常元素编辑允许持久化、只做元素级增量、删除始终软删除、已提交图片资产不随画布删除而删除。
