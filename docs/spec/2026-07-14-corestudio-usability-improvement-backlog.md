# CoreStudio 易用性优化待确认清单

> 所属项目：CoreStudio（工业设计助手）
>
> 状态：收集中，待用户确认
>
> 开发门禁：在用户明确说“可以开发了”之前，不创建实施分支、不编写实施计划、不修改功能代码。

## 这份文档解决什么问题

前一轮已经完成两个最紧急的边界收口：

- ACP 默认进入实验性功能，普通用户不再默认看到 ACP 配置、对话和调试入口。
- Codex 内置画布不再提供生成输入框，任务回到 Codex 对话中发起。

但当前产品仍然要求用户理解“Agent 集成、网页画布、Board、CLI、Local Bridge、环境变量”等技术概念。底层能力已经基本存在，缺少的是一个真正面向用户的安装、发现、连接、使用和修复闭环。

本清单只整理下一步需要改变的产品行为和待确认问题，不进入具体技术实现。

## 已确认的产品原则

1. **任务在哪里发起，哪里就是调度者。** CoreStudio 内发起的直接生成由 CoreStudio 调度；Codex 内发起的任务由 Codex 调度。
2. **Codex 默认使用自身生图能力。** 不通过 ACP 绕回 Codex，也不默认使用 CoreStudio 已配置的付费 API。
3. **CLI / Local Bridge 是后台数据通道。** 普通用户不需要学习它们，外部写回仍必须经过这条受控路径。
4. **ACP 有价值，但不是默认路径。** 它只用于从 CoreStudio 内部把任务交给外部 Agent，并继续作为手动开启的实验性功能。
5. **CoreStudio 可以提供自己的生图能力给 Agent。** 但必须由用户明确授权，不得静默切换，不得向 Agent 暴露 API Key。
6. **先完成易用性优化，再更新 Excalidraw 基线。** 基线同步不进入本轮范围。

## 当前已经完成的部分

- [x] ACP 新用户默认关闭。
- [x] 已配置并启用 ACP 的旧用户升级后保持可用。
- [x] 关闭 ACP 时保留配置、thread 和运行记录。
- [x] ACP 关闭时隐藏默认界面中的配置、状态、对话和调试入口。
- [x] Codex 内置画布移除生成输入框和 ACP 调度入口。
- [x] 产品文档明确 Codex、CoreStudio、CLI / Local Bridge 和 ACP 的职责边界。

## 需要改进的地方

### 一、把默认入口从“技术组件”改成“用户任务”

#### 当前现状

应用设置仍然把“网页画布、CLI、ACP Agent”解释成多条使用路径，并分别展示 Bridge、Board、CLI 状态。右下角状态浮层的主要动作仍然是“复制 CLI 环境变量”和“复制 Board 链接”。欢迎页也直接提供“Agent 集成”开关，并告诉用户通过网页画布和 CLI 连接项目。

#### 问题

- 普通用户仍需先理解技术架构，才能判断下一步做什么。
- “网页画布”和“CLI”看起来像两个需要分别配置的产品能力，实际上它们只是 Codex 协作背后的视觉与数据通道。
- ACP 虽然已经隐藏，但默认路径仍然没有收敛成一句清楚的用户语言。

#### 建议方向（待确认）

- 默认界面使用“在 Codex 中使用 CoreStudio”或“Codex 协作”，不再把 Board、CLI、Bridge 并列成主入口。
- 默认状态只回答三个问题：是否已安装、是否可以使用、如果不能使用下一步做什么。
- Board、CLI、Bridge、端口、token 和环境变量进入“连接详情”或“诊断信息”。
- 复制链接和环境变量从主操作降为高级诊断动作。
- 欢迎页不再直接暴露 Agent 技术开关，改为轻量的 Codex 协作状态或引导入口。

#### 已确认：移除应用菜单中的集成开关

- 移除 macOS 应用菜单中的“启用 Agent 集成”勾选项。
- 应用菜单继续保留“应用设置”入口，但不再直接修改集成启用状态。
- 集成的开启和关闭统一在应用设置中管理，避免菜单与设置同时承担配置职责。
- 右下角状态浮层只展示状态、诊断和快捷动作，不增加第二个启用开关。
- 移除菜单入口时不改变现有配置数据，也不静默修改用户当前的启用状态。

#### 已确认：大幅简化设置页的信息架构

当前设置页的问题不只是文案过多，而是仍然要求用户先理解产品内部结构：

- 顶部同时展示 Bridge、当前项目、网页画布、CLI 和 ACP 五类状态。
- 中间把网页画布、CLI 和 ACP Agent 解释成三条并列路径。
- 下方再次分别解释网页画布和 CLI，并提供复制链接、打开画布和复制环境变量等技术动作。
- 用户看完能够知道系统由哪些组件组成，但仍然不知道自己应该先做什么。

设置页后续按以下原则重构：

- 默认页面不再展示 Bridge、Board、CLI、ACP 的状态矩阵。
- 不再用三张说明卡教育用户理解网页画布、CLI 和 ACP 的关系。
- 不再重复展示“网页画布”和“CLI”独立说明区。
- 默认层只表达一个用户目标，例如“在 Codex 中使用 CoreStudio”。
- 每个状态只给一个清楚结论和一个主要动作，例如“安装 Codex 集成”“修复”“重新检查”或“请先打开项目”。
- 安装完成且当前可用时，只显示简洁的“已可用”状态，不继续铺开内部组件信息。
- Board、CLI、Bridge、端口、环境变量和 token 统一下沉到“连接详情”或“高级诊断”。
- 实验性外部 Agent 单独放在“实验性功能”，不与 Codex 默认协作路径并列。
- 设置页的目标是让用户完成配置和修复，不承担完整使用教程；详细原理和合同继续留在文档中。

建议的默认结构（方向已确认，具体文案与控件待后续确认）：

1. **Codex 协作**：总体状态、简短说明、唯一主要动作。
2. **连接详情**：默认收起，展示版本、CLI、Bridge、当前项目和诊断信息。
3. **实验性功能**：外部 Agent 独立开关；开启后才显示 ACP 配置。

模型 API 配置继续属于 CoreStudio 直接生成，不塞入 Codex 协作状态区。

### 二、提供一次性 Codex 集成安装闭环

#### 当前现状

- CLI 已包含在应用代码中，并能读取本机 session 文件自动发现当前 Bridge 和项目。
- 正式安装包还没有把 `corestudio` 安装为系统或 Codex 可稳定发现的命令。
- 仓库里没有可以实际分发的 CoreStudio Codex Skill、插件或对应安装器。
- 当前 README 仍明确要求源码环境使用 `node bin/corestudio.cjs`，正式安装不会提供全局命令。

#### 问题

“自动发现当前会话”只解决了 CLI 运行后的连接问题，没有解决 Codex 如何找到 CLI、理解命令、打开当前项目和完成写回的问题。

#### 建议方向（待确认）

CoreStudio 提供一个机器级的“安装 Codex 集成”动作，并形成完整状态机：

- 未安装
- 正在安装
- 已安装且可用
- 有更新
- 安装损坏或权限异常
- CoreStudio 未运行
- 当前没有打开项目
- 可以修复

安装内容至少包括：

- 稳定的 `corestudio` CLI 入口。
- Codex 可发现的 CoreStudio 使用说明和能力合同。
- 当前 session 自动发现。
- 安装版本和兼容性信息。
- 安装后的真实连接测试。
- 更新、修复和卸载能力。

### 三、跑通 Codex 中的自然语言主路径

#### 目标体验

用户在 Codex 中直接说：

- “打开当前 CoreStudio 项目。”
- “看看我现在选中的参考图。”
- “参考右边这张图优化当前设计。”
- “把刚生成的结果放回画布。”

Codex 应自动完成：

1. 判断 CoreStudio 是否运行、集成是否可用。
2. 发现当前项目；没有当前项目时，再提供最近项目供用户选择。
3. 在 Codex 内置浏览器中打开当前画布。
4. 读取选区、参考图原始路径和必要上下文。
5. 默认使用 Codex 自身能力生成或处理图片。
6. 通过 CLI / Local Bridge 写入图片、生成记录和画布元素。
7. 返回可以定位和继续修改的结果。

#### 需要补齐的状态

- CoreStudio 没有运行。
- Codex 集成没有安装。
- 集成版本不兼容或需要更新。
- Agent 集成被关闭。
- 没有打开项目。
- 当前选区没有可用图片。
- 原图路径不可读。
- 写回被拒绝、超时、冲突或只完成了一部分。
- 写回成功，但结果在画布中无法定位。

### 四、重新设计连接状态和错误反馈

#### 当前现状

设置页和状态浮层展示大量底层状态，但没有形成统一的用户结论。例如 CLI 显示“可自动发现当前会话”，用户仍可能不知道 Codex 是否真的能调用它。

#### 建议方向（待确认）

- 最上层只显示“Codex 协作可用 / 需要安装 / 需要修复 / 请打开项目”。
- 每个不可用状态只给一个主要动作，例如“安装”“重新检查”“打开项目”“修复集成”。
- 展开诊断后再显示 CLI 路径、Bridge 地址、session 时间、项目 token 状态和版本信息。
- 区分“安装成功”和“当前连接成功”，避免安装状态、运行状态、项目状态混成一个开关。
- 支持复制一份不含密钥的诊断信息，方便交给 Codex 或开发者排查。

#### 已确认：简化并重新定位状态浮窗

当前右下角 Agent 状态浮窗同时展示连接结论、ACP Agent、当前项目、项目路径、本地桥、CLI、内置浏览器以及五个快捷动作，信息量过大，也继续强化了用户对底层组件的关注。

后续调整原则已经确认：

- 状态浮窗不再承担集成开关或配置职责，集成启停只在应用设置中管理。
- 浮窗从“大型 Agent 控制面板”改成轻量状态入口，只帮助用户判断当前是否可用以及下一步做什么。
- 默认不展示 ACP Agent 卡片、完整项目路径、本地桥地址、CLI 状态和 Board 状态等技术详情。
- “复制 CLI 环境变量”和“复制 Board 链接”不再作为默认快捷动作；如仍需保留，进入连接详情或高级诊断。
- 状态应自动更新，普通用户不应依赖“刷新状态”才能知道当前是否可用。
- “打开设置”可以保留为异常状态下的处理入口，但不与多个技术动作并列。
- 浮窗在界面中的具体位置需要重新评估，避免遮挡画布或形成过强的固定入口；新位置尚待确认。

建议的最小信息结构（待确认）只保留：

- 总体状态：可用、需要安装、需要修复、请打开项目。
- 当前项目名称；没有项目时给出明确提示。
- 一个与当前状态对应的主要动作，例如“打开设置”“安装”“修复”或“打开项目”。
- 可选的“连接详情”，用于展开技术诊断信息。

### 五、提供经过授权的 CoreStudio 生图 Agent 能力

#### 已确认边界

- 默认关闭。
- 只有 CoreStudio 已配置可用模型且用户明确授权时，才向 Agent 声明该能力。
- 用户明确说“使用 CoreStudio 模型”时才调用。
- Codex 自身生图不可用时，不得静默切换到可能产生费用的 CoreStudio API。
- API Key 始终留在 CoreStudio。
- CoreStudio 负责调用模型、保存结果和建立生成记录；CLI 只返回结构化状态和结果。

#### 仍需确认

- Agent 只能使用 CoreStudio 当前默认模型，还是可以在已配置模型中选择。
- 每次调用都确认，还是首次授权后仅在可能产生费用或切换模型时确认。
- 长时间生成采用同步等待还是异步任务；至少需要进度、取消、失败和结果查询。
- Agent 生成结果是否自动放入画布，还是允许仅保存为生成记录后由用户确认插入。

### 六、继续收口 ACP 的实验性体验

ACP 的默认隐藏已经完成，但开启后的体验仍然偏工程配置：命令、参数、工作目录、任务模板和 run log 都需要用户理解。

建议后续只在用户主动开启实验性功能后处理：

- 使用面向任务的名称解释它，例如“连接外部 Agent（实验性）”，ACP 缩写放在说明中。
- 优先提供已知 Agent 预设和连接验证，再提供自定义命令。
- 明确区分“实验功能已开启”“外部 Agent 已配置”“外部 Agent 当前可用”。
- 调试 JSON 和 run log 继续留在高级调试，不进入普通对话路径。
- 不把 ACP 安装与 Codex 默认集成安装合并成同一个开关。

### 七、简化模型 API 的首次配置

CoreStudio 直接生成仍然是默认主路径，但当前配置以服务商、Key、模型 ID 和能力参数为中心。

需要继续收集和确认：

- 首次生成时是否提供最小配置引导，而不是直接展开完整高级设置。
- 默认只要求选择服务商、填写 Key、选择模型；自定义模型能力、adapter、尺寸模式等进入高级设置。
- 保存后是否立即执行轻量连接检查，并明确区分“Key 已保存”和“模型可用”。
- 配置失败时是否给出服务商可理解的处理建议，而不是只展示原始错误。
- 是否需要把“CoreStudio 直接生成”和“连接外部 Agent”放在同一初始选择中；当前建议是不放在一起，ACP 继续留在实验性功能。

### 八、统一命名和文案层级

默认用户界面建议减少或隐藏以下术语：

- Agent Board / Board 链接
- Local Bridge / Bridge
- CLI 环境变量
- project token
- ACP

这些术语仍应保留在开发文档、CLI contract 和诊断详情中。

默认界面优先使用：

- Codex 协作
- 当前项目
- 打开画布
- 读取当前选择
- 写回结果
- 安装 / 更新 / 修复
- 连接详情
- 实验性外部 Agent

## 建议优先级（待确认）

### 第一组：让普通用户不用复制任何东西

1. 默认入口和文案重组。
2. Codex 集成安装、更新、修复和卸载。
3. 稳定 CLI 入口和 Codex 能力说明。
4. 当前项目与 session 自动发现。
5. Codex 自然语言打开画布和读写回主路径。

### 第二组：让异常状态可以自己恢复

1. 统一安装、运行、项目和兼容性状态。
2. 提供明确的下一步动作。
3. 增加安全诊断信息和连接验证。
4. 覆盖写回失败、冲突和结果不可定位。

### 第三组：补充高级能力

1. 授权后的 CoreStudio 生图 Agent 能力。
2. ACP 开启后的连接和配置体验。
3. 模型 API 首次配置和高级设置分层。

- 允许 Agent 直接读取 API Key。
- 允许 Agent 绕过 CLI / Local Bridge 修改项目文件。
- 未经同意自动切换到可能收费的 CoreStudio API。
- 在需求确认阶段修改功能代码或提前创建实施分支。
